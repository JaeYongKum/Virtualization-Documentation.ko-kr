# <a name="containers-vs-virtual-machines"></a>컨테이너 대 가상 컴퓨터

이 항목에서는 컨테이너와 Vm (가상 컴퓨터)의 주요 유사점과 차이점에 대해 설명 하 고 각각을 사용할 수 있습니다. 각 컨테이너와 Vm에는 각각 사용이 포함 되며, 특히 클라우드에서 컨테이너를 실행 하는 경우에는 하드웨어에서 직접 실행 하는 대신 Vm을 호스트 운영 체제로 사용 하는 경우가 많습니다.

컨테이너에 대 한 개요는 [Windows 및 컨테이너](index.md)를 참조 하세요.

## <a name="container-architecture"></a>컨테이너 아키텍처

컨테이너는 호스트 운영 체제에서 응용 프로그램을 실행 하기 위한 격리 된 경량 사일로 (silo)입니다. 컨테이너는 호스트 운영 체제의 커널 (운영 체제의 덮여 있는 것으로 간주할 수 있음) 위에 빌드 되며,이 다이어그램에 표시 된 대로 사용자 모드에서 실행 되는 앱과 일부 간단한 운영 체제 Api 및 서비스만 포함 합니다.

![컨테이너를 커널을 위에서 실행 하는 방법을 보여 주는 아키텍처 다이어그램](media/container-diagram.svg)

## <a name="virtual-machine-architecture"></a>가상 컴퓨터 아키텍처

컨테이너와 대조적으로 Vm은이 다이어그램에 표시 된 대로 자체 커널을 포함 하 여 완전 한 운영 체제를 실행 합니다.

![Vm이 호스트 운영 체제 옆의 전체 운영 체제를 실행 하는 방법을 보여 주는 아키텍처 다이어그램](media/virtual-machine-diagram.svg)

## <a name="containers-vs-virtual-machines"></a>컨테이너 대 가상 컴퓨터

다음 표에서는 이러한 보완적인 기술의 유사점과 차이점을 보여 줍니다.

|                 | 가상 컴퓨터  | 컨트롤러  |
| --------------  | ---------------- | ---------- |
| 격리가       | 호스트 운영 체제 및 기타 Vm에서 완벽 하 게 격리할 것을 제공 합니다. 이는 같은 서버나 클러스터에서 경쟁 회사의 앱을 호스팅과 같은 강력한 보안 경계가 중요할 때 유용 합니다. | 일반적으로 호스트 및 다른 컨테이너와의 간편한 격리를 제공 하지만 VM으로 강력 하 게 보안 경계를 제공 하지는 않습니다. (경량 VM의 각 컨테이너를 격리 시키기 위해 [hyper-v 격리 모드](../manage-containers/hyperv-container.md) 를 사용 하 여 보안을 높일 수 있습니다.) |
| 운영 체제 | 커널을 비롯 한 전체 운영 체제를 실행 하 여 시스템 리소스 (CPU, 메모리 및 저장소)가 더 필요 합니다. | 운영 체제의 사용자 모드 부분을 실행 하 고, 시스템 리소스를 적게 사용 하 여 앱에 필요한 서비스만 포함 하도록 조정할 수 있습니다. |
| 게스트 호환성 | 가상 컴퓨터 내부의 모든 운영 체제에 대해서만 실행 됩니다. | [호스트와 동일한 운영 체제 버전](../deploy-containers/version-compatibility.md) 에서 실행 (hyper-v 격리를 사용 하 여 간단한 VM 환경에서 이전 버전의 동일한 OS를 실행할 수 있음)
| 배포     | Windows 관리 센터 또는 Hyper-v 관리자를 사용 하 여 개별 Vm을 배포 합니다. PowerShell 또는 System Center Virtual Machine Manager를 사용 하 여 여러 Vm을 배포 합니다. | 명령줄을 통해 Docker를 사용 하 여 개별 컨테이너를 배포 합니다. orchestrator를 Azure Kubernetes Service와 같이 사용 하 여 여러 컨테이너를 배포 합니다. |
| 운영 체제 업데이트 및 업그레이드 | 각 VM에 운영 체제 업데이트를 다운로드 하 고 설치 합니다. 새 운영 체제 버전을 설치 하려면 업그레이드 하거나 완전히 새로운 VM을 만드는 것이 좋습니다. 특히 Vm이 많은 경우에는 시간이 오래 걸릴 수 있습니다. | 컨테이너 내에서 운영 체제 파일을 업데이트 하거나 업그레이드 하는 것은 동일 합니다. <br><ol><li>최신 버전의 Windows 기본 이미지를 가리키도록 컨테이너 이미지의 빌드 파일 (Dockerfile 이라고 함)을 편집 합니다. </li><li>이 새로운 기본 이미지를 사용 하 여 컨테이너 이미지를 다시 작성 합니다.</li><li>컨테이너 이미지를 컨테이너 레지스트리에 푸시합니다.</li> <li>Orchestrator를 사용 하 여 다시 배포 합니다.<br>Orchestrator는 배율에서이 작업을 수행 하는 강력한 자동화 기능을 제공 합니다. 자세한 내용은 [자습서: Azure Kubernetes 서비스에서 응용 프로그램 업데이트](https://docs.microsoft.com/azure/aks/tutorial-kubernetes-app-update)를 참조 하세요.</li></ol> |
| 영구 저장소 | 단일 VM에 대 한 로컬 저장소에는 VHD (가상 하드 디스크)를 사용 하거나 여러 서버에서 공유 하는 저장소에 대 한 SMB 파일 공유 | 단일 노드에 대 한 로컬 저장소 또는 여러 노드나 서버에서 공유 하는 저장소에 대 한 Azure 파일 (SMB 공유)에 Azure 디스크를 사용 합니다. |
| 부하 분산 | 가상 컴퓨터 로드 균형 조정은 장애 조치 클러스터의 다른 서버에서 실행 중인 Vm을 이동 합니다. | 컨테이너 자체는 이동 하지 않습니다. 대신 orchestrator는 클러스터 노드에서 컨테이너를 자동으로 시작 하거나 중지 하 여 부하와 가용성의 변화를 관리할 수 있습니다. |
| 내결함성은 | Vm의 운영 체제가 새 서버에서 다시 시작 되는 클러스터의 다른 서버로 Vm을 장애 조치할 수 있습니다.  | 클러스터 노드에 장애가 발생 하는 경우이를 실행 하는 모든 컨테이너는 orchestrator에 의해 다른 클러스터 노드에 빠르게 다시 만들어집니다. |
| 네트워킹     | 가상 네트워크 어댑터를 사용 합니다. | 가상 네트워크 어댑터의 격리 된 보기를 사용 하 여 가상화를 제공 하는 반면 호스트의 방화벽은 컨테이너와 공유 되지만 리소스를 덜 사용 합니다. 자세한 내용은 [Windows 컨테이너 네트워킹](../container-networking/architecture.md)을 참조 하세요. |
